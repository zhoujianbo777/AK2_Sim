## 1. 概述

### 1.1 目的与背景

本文档描述基于 AK2（DJ628.27）超声波感知系统的 **PC 端算法仿真软件**的设计方案。该软件运行于 Windows 操作系统下，以已采集完成的实车历史测试数据为输入，通过调用经过训练的 AI 模型推理库及传统算法模块，对超声波传感器的感知数据进行回放、处理与可视化，实现在不依赖实车硬件的条件下完成算法验证、性能分析及 AI 与传统算法对比评估的目标。

本软件定位于**研发阶段的算法调试与验证工具**，是 AI 模型从 PC 端训练结果到 ECU 端嵌入式部署之间的关键验证环节。

### 1.2 适用范围

本软件适用于以下使用场景：

| 场景 | 说明 |
|------|------|
| 算法回归测试 | 对新版本模型或算法进行批量历史数据回放，统计性能指标 |
| 可视化调试 | 对特定场景数据帧进行逐帧回放，观察算法内部中间状态 |
| AI vs 传统算法对比 | 同一数据集在两套算法下并行运行，直观展示 AI 的感知优势 |
| 边界案例分析 | 针对采集的极端场景（悬空障碍物、湿滑地面等）验证 AI 新增能力 |

### 1.3 设计约束

- **运行平台**：Windows 10/11，x86/x64 架构
- **开发语言**：Python（不含嵌入式推理核心，推理核心由预编译动态库提供）
- **推理一致性**：PC 端调用与 ECU 端相同的推理逻辑（编译为 x64 动态库版本），确保仿真结果与嵌入式运行结果一致
- **数据来源**：全部使用存储于 `TestData/` 目录下的历史采集数据，不依赖实时硬件连接
- **文档约束**：本文档不包含任何代码或伪码，仅描述功能、接口、流程与设计决策

## 2. 系统架构设计

### 2.1 总体架构

PC 端仿真软件由以下五个顶层模块组成，各模块之间通过内部消息队列和共享数据结构进行解耦通信：

```
┌─────────────────────────────────────────────────────────────┐
│                     PC 端仿真软件总体架构                     │
├───────────────┬──────────────────────────────────────────────┤
│  数据管理模块  │  负责历史数据文件的加载、解析、帧序列管理        │
├───────────────┼──────────────────────────────────────────────┤
│  算法引擎模块  │  包含：传统算法引擎 / AI算法引擎（调用动态库）   │
├───────────────┼──────────────────────────────────────────────┤
│  结果融合模块  │  对算法输出进行后处理，生成可视化所需的结构化结果 │
├───────────────┼──────────────────────────────────────────────┤
│  图形显示模块  │  多窗口 GUI，包含包络图/OGM/统计面板等子窗口    │
├───────────────┼──────────────────────────────────────────────┤
│  对比控制模块  │  协调两个程序实例的同步运行和结果对比            │
└───────────────┴──────────────────────────────────────────────┘
```

### 2.2 单进程双引擎架构

本软件在**单个进程**内同时运行传统算法引擎和 AI 算法引擎，两套引擎共享同一个数据管理模块和同一个回放定时器，天然保证帧序一致性，无需任何进程间通信机制。

```
单进程（sim_main.py）
│
├── 数据管理模块（读取 TestData/，共享）
│
├── 传统阈值分类引擎（engine_trad）
│   └── 结果融合模块（result_fusion_trad）
│
├── AI 推理引擎（engine_ai，调用 inference_core_x64.dll）
│   └── 结果融合模块（result_fusion_ai）
│
├── W1 主控制面板（共享，居中）
├── 传统算法显示面板（DisplayWindow, mode="traditional"，左侧）
└── AI 算法显示面板（DisplayWindow, mode="ai"，右侧）
```

每个定时器 tick（10Hz）中，主控制器依次调用两套引擎处理同一帧数据，分别通知各自的显示面板刷新，实现**同帧同屏**的并排对比。

### 2.3 与 ECU 推理库的关系

```
ECU 嵌入式端                    PC 仿真软件端
─────────────────               ───────────────────────────
inference_core.c  ──编译──→  inference_core_x64.dll
（ARM Cortex-R52）              （x86/x64 Windows DLL）
        │                               │
  ECU固件调用                    Python ctypes 调用
        │                               │
  相同数值逻辑                   相同数值逻辑（可复现性保证）
```

PC 端 Python 脚本通过 `ctypes` 接口加载 `inference_core_x64.dll`，调用与 ECU 端完全相同的推理函数入口，确保两端推理结果在数值上等效（精度误差 ≤ 0.01）。

## 3. 测试数据管理模块设计

### 3.1 TestData 目录结构约定

软件要求历史测试数据存储于程序根目录下的 `TestData/` 文件夹，目录结构如下：

```
TestData/
├── session_YYYYMMDD_NNN/        # 每次采集会话（按日期+序号命名）
│   ├── esi_frames.bin           # ESI总线原始帧（EDI+包络，二进制格式）
│   ├── can_signals.csv          # CAN总线同步信号（车速、转向角、档位等）
│   ├── session_meta.json        # 采集元数据（天气、温度、车型、路面类型等）
│   └── ground_truth.json        # 可选：人工标注的GT标签（用于评估模式）
├── session_YYYYMMDD_NNN/
│   └── ...
└── index.json                   # 所有会话的索引文件（供软件快速加载列表）
```

### 3.2 数据加载器功能设计

数据管理模块提供以下核心功能：

#### 3.2.1 会话选择

软件启动后，自动扫描 `TestData/` 目录，从 `index.json` 加载所有可用会话列表，在界面侧边栏以树形结构展示，支持按日期、场景类型、天气条件进行筛选。

#### 3.2.2 帧序列管理

将 `esi_frames.bin` 解析为帧对象序列，每帧包含：

| 字段 | 来源 | 说明 |
|------|------|------|
| 帧时间戳（ms） | ESI帧头 | 相对于会话开始时间的偏移量 |
| 12路 EDI 数据 | EDI帧解析 | 每路：距离、幅度、置信度、回波类型 |
| 12路归一化包络序列 | 包络帧解析 | 每路256点，float32，已归一化 |
| 同步CAN信号 | can_signals.csv 时间对齐 | 车速、转向角、档位 |

#### 3.2.3 回放控制

数据管理模块向上层提供以下回放控制能力，由 GUI 控制条触发：

| 控制操作 | 说明 |
|---------|------|
| 播放 / 暂停 | 按照原始采集帧率（10Hz）连续回放，或暂停在当前帧 |
| 逐帧步进 | 向前/向后各步进一帧，用于精细调试 |
| 跳转到指定帧 | 输入帧编号直接跳转 |
| 速率调节 | 支持 0.25×、0.5×、1×、2×、5× 回放速率 |
| 循环播放 | 到末帧后自动跳回首帧 |
| 区间播放 | 设置起止帧编号，仅回放指定区间 |

#### 3.2.4 批量评估模式

除交互式回放外，软件还支持**批量评估模式**：对整个会话或多个会话的全部帧自动完成算法处理，最终输出统计报告（精确率、召回率、混淆矩阵等），该模式不显示逐帧动画，仅在完成后展示汇总结果。

---

### 3.3 噪声注入与鲁棒性测试模式

为在无实车硬件的条件下验证模型对劣化输入数据的鲁棒性，数据管理模块提供**在线噪声注入**功能：在数据帧送入算法引擎之前，对原始数据实时叠加可配置的干扰信号，然后观察算法输出的变化。噪声注入功能通过 W1 主控制面板的"鲁棒性测试"子面板进行配置，与正常回放模式不互斥（可随时开启/关闭）。

#### 3.3.1 可配置噪声类型

| 噪声类型 | 作用目标 | 可配置参数 | 物理含义 |
|---------|---------|---------|---------|
| 高斯白噪声叠加 | 包络序列（所有12路） | 信噪比（20~50dB可调） | 模拟电磁干扰或ADC量化噪声恶化 |
| 单路传感器完全故障 | 指定路号（0~11） | 故障路序号（单选/多选） | 验证SDK降级模式自动激活行为 |
| 多路传感器随机丢帧 | ESI帧有效性标志 | 丢帧概率（0~50%） | 模拟ESI总线偶发通信错误 |
| 温度漂移模拟（包络时延偏移） | 包络时间轴 | 偏移量（±10采样点） | 模拟传感器在极端温度下的频率漂移 |

> **注意**：噪声注入不修改 `TestData/` 原始文件，仅在内存中对当前帧数据临时处理，关闭噪声注入后立即恢复原始数据。

#### 3.3.2 鲁棒性测试批量模式

鲁棒性测试支持**批量自动化**运行：对测试数据集中所有帧依次注入上述各类噪声，记录推理结果变化（分类结果是否翻转、置信度变化量），自动生成鲁棒性测试报告，定量评估模型对各类干扰的容忍阈值。报告格式与第7.3节批量评估报告一致，额外增加"噪声参数-精度下降曲线"图表。

## 4. 算法引擎模块设计

### 4.1 算法引擎总览

软件内置两套独立的算法引擎，可分别在传统实例和 AI 实例中运行：

| 引擎类型 | 显示面板 | 核心方法 | 说明 |
|---------|---------|---------|------|
| 传统阈值分类引擎 | 传统算法显示面板（左侧） | 基于幅度阈值 + 距离规则的规则系统 | 代表现有方案的感知能力上限 |
| AI 推理引擎 | AI 算法显示面板（右侧） | 调用编译好的 inference_core_x64.dll | 运行与 ECU 一致的 M1→M2-SC→M2-SA 模型链路 |

### 4.2 传统阈值分类引擎设计

传统引擎采用基于规则的分级判决方式，不依赖任何机器学习模型，完全由 Python 层实现：

#### 4.2.1 有效性判决

对每路 EDI 的置信度和幅度进行双门限判决：
- 一级门限：置信度高于固定阈值（50）且幅度高于噪声基底（滑动窗口均值 × 3σ）
- 二级门限：连续3帧均满足一级门限才确认为有效目标（去抖动）

#### 4.2.2 障碍物分类（简化）

传统引擎仅支持简化的 **5类**分类（对应 AK2 前代 AK1 方案）：

| 类别ID | 描述 | 判别规则 |
|--------|------|---------|
| 0 | 硬质障碍物 | 幅度高（>阈值H），包络单峰，峰值陡峭 |
| 1 | 软质/小型障碍物 | 幅度中等（阈值L~H），包络宽泛 |
| 2 | 空旷/无障碍 | 幅度低于噪声地板 |
| 3 | 地面杂波 | 包络拖尾长（>20点）且峰值对应距离<0.3m |
| 4 | 未知 | 不满足以上任何规则 |

**传统引擎的已知局限**（将在对比中显现）：
- 无法区分悬空障碍物与普通墙体
- 无法识别路沿等低矮障碍物
- 在湿滑地面上误报率高（镜面反射导致异常幅度）
- 无材质信息输出

### 4.3 AI 推理引擎设计

AI 推理引擎通过 Python 的 `ctypes` 机制加载 `inference_core_x64.dll`，该动态库由 ECU 端同名推理核心代码交叉编译（目标架构改为 x86_64 Windows）而来，确保推理逻辑与 ECU 端完全一致。

#### 4.3.1 动态库加载与接口映射

`inference_core_x64.dll` 暴露的 C 接口与 ECU SDK 中定义的接口完全相同，PC 端 Python 层仅负责数据格式转换（Python 原生类型 → ctypes 缓冲区）和结果回收，不做任何算法逻辑修改。

#### 4.3.2 AI 推理链路

AI 引擎按以下顺序依次调用推理库中的各模块：

```
输入帧数据（EDI特征 + 包络序列）
    │
    ▼
[步骤1] 特征预处理：z-score归一化 / 包络标准化（对应SDK第4章）
    │
    ▼
[步骤2] M1推理：障碍物有效性检测（12路独立输出）
    │
    ▼
[步骤3] M2-SC推理：9类障碍物分类 + 材质硬度评分 + 悬空高度估算
    │         （仅对 p_valid ≥ 0.5 的通道执行，输出中间特征 feat[N×32]）
    │
    ▼
[步骤4] M2-SA推理：组内空间注意力精化（仅当 m2_sa_enabled=true）
    │         （前/后组各独立处理，输出精化后 9类概率）
    │
    ▼
[步骤5] 融合决策：置信度加权、12×9矩阵融合（对应SDK第6章）
    │
    ▼
输出结构化结果（每路：类别ID、9类概率、SC原始结果、材质评分、净高估算）
```

#### 4.3.3 推理结果缓存

每帧的推理结果被保存至内存缓冲区（固定大小的环形队列，默认保留最近200帧）。用户在 GUI 中逐帧步进时，可从缓存中直接回取历史结果，无需重复推理，提升响应速度。

## 5. 图形化显示模块设计

### 5.1 显示窗口总体布局

软件采用多窗口设计，每个运行实例拥有独立的一组窗口。各窗口均可独立调整大小、停靠或浮动，支持 Windows 多显示器跨屏布局。

#### 5.1.1 单实例窗口组成（以AI实例为例）

| 窗口编号 | 窗口名称 | 尺寸建议 | 主要内容 |
|---------|---------|---------|---------|
| W1 | 主控制面板 | 420×1275 | 会话选择、回放控制条、速率调节、当前帧信息 |
| W2 | 包络波形视图 | 与W3/W6同宽 | 12路传感器包络序列波形图（2行×6列排列） |
| W3 | 障碍物分类视图 | 与W2/W6同宽 | 12路当前帧的分类结果柱状图；启用 M2-SA 时显示 SC/SA 双柱对比 |
| W4 | 占据栅格地图（OGM） | 与W1同高 | 俧视角度的车辆+障碍物空间分布图 |
| W6 | 性能统计面板 | 与W2/W3同宽 | 当前会话的实时统计（精确率、召回率、混淤矩阵） |

#### 5.1.2 显示面板布局示意图

W2、W3、W4、W6 整合在一个显示面板窗口中，初始高度与 W1 主控面板等高（1275px）：

```
┌───────────────┐   ┌─────────────────────┬───────────────┐
│               │   │  W2 包络波形视图     │               │
│  W1 主控制    │   ├─────────────────────┤  W4 OGM       │
│  面板         │   │  W3 障碍物分类视图  │  占据栅格地图  │
│               │   ├─────────────────────┤  （与W1等高） │
│  420×1275px   │   │  W6 性能统计面板    │               │
└───────────────┘   └─────────────────────┴───────────────┘
                     W2 / W3 / W6 同宽左列，W4 全高右列
```

### 5.2 各窗口详细设计

#### 5.2.1 W1：主控制面板

主控制面板是用户操作的核心界面，分为三个区域：

**上区：会话管理**
- 显示当前已加载会话的名称、采集时间、帧总数、场景类型（从session_meta.json读取）
- "打开数据集"按钮：弹出文件树，选择 TestData/ 下的会话目录
- "批量评估"按钮：切换至批量评估模式

**中区：回放控制条**
- 进度条：显示当前帧位置（可点击跳转）
- 时间码：格式为 `HH:MM:SS.mmm`（相对于会话开始时间）
- 播放/暂停、上一帧、下一帧、跳转输入框
- 速率选择下拉框（0.25×/0.5×/1×/2×/5×）
- 循环/区间播放开关

**下区：当前帧元数据**
- 显示当前帧的：帧编号、时间戳、车速（km/h）、转向角（°）、档位、温度

#### 5.2.2 W2：包络波形视图

W2 以 **2行×6列**（前排 S01~S06 一行、后排 S07~S12 一行）的网格排列，展示12路传感器的包络波形，每个子图包含：

- **X轴**：采样点索引（0~255），对应时间/距离维度
- **Y轴**：归一化幅度（0~1）
- **波形线**：当前帧的包络序列，用蓝色实线绘制
- **峰值标注**：在包络的主峰位置叠加红色三角形标记，并标注对应的距离值（m）
- **有效性背景色**：每个子图的背景色根据 **M1 MLP 输出的 `valid_flags`** 着色：`valid_flags ≥ m1_validity_threshold`（默认 0.5）时显示橙色背景（表示该探测区域内判定有障碍物），否则背景透明无色（表示无障碍物）。障碍物类型则不反映在背景，而是通过 W4 OGM 圆点颜色呈现。
- **传感器标签**：子图左上角显示传感器编号（S01~S12）和安装位置（如"左前1"）
- **历史轨迹**：同时叠加显示前5帧的半透明包络（用于观察包络变化趋势）

**有效性背景颜色编码**：

| 状态 | 背景色 | 说明 |
|------|--------|------|
| `valid_flags ≥ threshold` | 橙色半透明 | 该路探测区域内判定有障碍物 |
| `valid_flags < threshold` | 透明（无填充） | 该路探测区域内无障碍物 |

闈值小过 `config.yaml` 中的 `ai.m1_validity_threshold`（默认 0.5）配置。

#### 5.2.3 W3：障碍物分类视图

W3 以柱状图方式呈现 12 路传感器当前帧的分类结果，分前组（S01~S06）和后组（S07~S12）两行排列。

**基本显示（m2_sa_enabled: false 或传统算法实例）**

- X轴：组内传感器编号（每行 6 根柱）
- Y轴：当前帧该路的主分类置信度（0~1）
- 柱体颜色：对应 9 类统一颜色编码（见上表）

- 柱体内嵌（AI模式）：材质硬度 `H:0.xx`（置信度 ≥ 0.30 且非 Open 类时显示）；悬空净高 `⇑x.xxm`（Overhead 类时显示）
- 当前选中路（鼠标单击切换）：柱体以红色加粗边框高亮

**M2-SA 对比显示（m2_sa_enabled: true，AI 实例专属）**

启用空间注意力模块后，每路由单柱扩展为**前后并排双柱**：

```
     S01       S02       S03  …
  [SC][SA]  [SC][SA]  [SC][SA]
```

- **SC 柱**（左柱）：M2-SC 骨干单路独立输出，颜色饱和度降低至 60%，表示"精化前"
- **SA 柱**（右柱）：M2-SA 空间注意力精化后输出，全饱和颜色，表示"最终结论"
- **差异高亮**：当 SA 与 SC 的主类别不同时，SA 柱加橙色加粗边框，柱顶叠加 delta 标签（格式：`SC类→SA类`，如 `Cl→Cu`）
- **置信度箭头**：当 SA 与 SC 类别相同但置信度差值 > 0.10 时，在 SA 柱顶显示 `↑` 或 `↓` 小符号

启用空间注意力模块后，W3 始终保持 **2 行布局**（前组 + 后组），每个行标题后追加 `▏SA` 标识。SA 对比信息内联在单条柱上，而非添加额外行：

- 当 SA 修正了 SC 的分类结果时：柱上加橙色加粗边框，柱顶显示 `SC类→SA类` delta 标签（如 `Cl→Cu`）
- 当 SA 与 SC 类别相同但置信度差值 > 0.10 时：柱顶显示 `↑`/`↓` 小符号
- `m2_sa_enabled: false` 时，标题不显示 SA 标识，相同 2 行布局不变

#### 5.2.4 W4：占据栅格地图（OGM）

W4 展示俯视角度的空间感知结果，是最直观的感知可视化窗口：

**地图坐标系**：
- 以本车中心为原点
- 上方为车辆前进方向（正前方）
- 显示范围：前后各 **8m**，左右各 **6.5m**（与 `config.yaml` `display.ogm_range_front/side` 一致）
- 分辨率：每格 0.1m × 0.1m（100mm 精度）
- 网格尺寸：160 行（纵）× 130 列（横）

**显示内容**（由远及近的层次叠加）：
1. **栅格背景**：M6 OGM 占用概率用灬度色渲染栅格背景
2. **模型后处理**（依次应用）：
   - 前/后传感器组 `valid_flags` 门控：前6路均値<0.05 则前半 OGM 置零，>0.30 则全颟；线性过渡
   - 全局噪底阈値： OGM 小于 0.18 的格子置零
   - 平滑：`gaussian_filter(sigma=1.0)`（消除块状锟齿伪影）
3. **车辆轮廓**：用蓝色矩形线框表示车辆当前占据轮廓（尺寸从配置文件读取）
4. **障碍物标记**：每路传感器检测到的有效目标以彩色圆点表示（颜色编码同 W3）；启用 M2-SA 时，SA 修正了 SC 类别的通道在圆点外叠加白色细环
5. **传感器覆盖标视区——“Show Sensor FOV Sectors”开关**：12路传感器的探测方向和有效距离以半透明扇形叠加，`valid_flags ≥ m1_validity_threshold` 时对应桷形填充橙色
6. **轨迹历史**：本车历史轨迹（最近30帧）以灰色虚线绘制

**交互功能**：
- 悬停提示：鼠标悬停在障碍物标记上，浮窗显示该障碍物的详细信息（类别、距离、置信度、材质评分；启用 M2-SA 时额外显示 SC 原始类别）

#### 5.2.5 W6：性能统计面板

W6 在回放过程中实时累积统计数据。面板行为取决于当前 session 是否携带有效的标注数据（`ground_truth.json` 内 `frame_labels` 非空）：

**有标注数据时（训练 session）**

**实时指标（左区）**：
- 已处理帧数 / 总帧数（进度条格式：`Accuracy: XX%  (M/N frames)`）
- 当前累积的各类别精确率、召回率、F1（表格形式，随回放实时刷新）；启用 M2-SA 时显示 SC / SA 两列对比，便于直观评估注意力模块的逐类别增益
- 总体准确率进度条（绿色填充）

**混淆矩阵热力图（右区）**：
- 统一采用 9×9 混淆矩阵（传统算法通过 `TRAD_TO_UNIFIED` 映射到同一类别空间）
- 启用 M2-SA 时，右区并列展示两个矩阵：左为 SC 骨干混淆矩阵，右为 SA 精化混淆矩阵，标题分别标注"SC"/"SA"
- 热力图着色：对角线趋绿（正确分类），非对角线趋红（误分类）；颜色深浅反映行归一化百分比（0–100%）
- 每格标注百分比数值（与背景色一致的归一化比例），正确率 ≥ 50% 时用黑色字体，否则用白色字体
- 混淆矩阵每 500 ms 最多刷新一次，避免频繁 matplotlib 渲染阻塞主线程

**无标注数据时（测试 session）**

加载无标注 session 后，W6 立即进入"不可评估"状态：
- 进度条显示"**无标注数据 — 本 session 不可评估**"，不显示帧计数
- 各类别指标表格恢复为占位符"—"，不显示零值（避免与"准确率为 0%"混淆）
- 混淆矩阵区域显示"**无标注数据**"文字提示

> **区分原则**：进度条显示 `0% (0/N frames)` 表示 session 有标注但尚未回放；显示"无标注数据"则表示该 session 本身不具备评估条件，两者语义不同。

**"导出报告"按钮**：将当前统计结果导出为 Markdown 格式的性能报告文件（无标注数据时导出内容为空表）。

## 6. 双引擎对比运行机制设计

### 6.1 架构说明

两套算法引擎在**同一进程**内并行运行，由主控制器（`AK2SimApp`）统一驱动。每个定时器 tick 中，主控制器从数据管理模块取得当前帧后，依次调用传统引擎和 AI 引擎各处理一次，并分别通知对应的显示面板刷新，天然保证两侧显示内容为同一帧数据，无需任何帧同步机制。

用户通过根目录下的 `launch_compare.bat` 或直接运行 `python sim_main.py` 即可启动包含双引擎的完整对比界面，无需额外操作。

### 6.2 窗口布局

软件启动后，三个独立窗口自动排列于主屏幕（1920×1080 或更高分辨率推荐）：

```
┌──────────────────────┬────────────┬──────────────────────┐
│  传统算法显示面板     │  W1        │  AI 算法显示面板      │
│                      │  主控制    │                      │
│  W2 包络波形视图      │  面板      │  W2 包络波形视图      │
│  W3 分类视图          │  （共享）  │  W3 分类视图          │
│  W4 OGM              │  420px宽   │  W4 OGM              │
│  W6 统计面板          │            │  W6 统计面板          │
└──────────────────────┴────────────┴──────────────────────┘
 （屏幕剩余宽度各占一半）
```

W1 主控制面板居中显示，宽度固定 420px，两侧显示面板平分剩余屏幕宽度。W1 的所有回放控制操作（播放/暂停/跳帧/速率/会话切换）同时作用于两套引擎，用户操作一次即可同步控制双侧。

### 6.3 AI 优势场景高亮标注

对比运行时，主控制器在每帧渲染后实时比较两套引擎的分类结果。当 AI 实例正确识别而传统算法实例误判或漏检时，在 AI 侧的 W4 OGM 窗口顶部显示一条橙色高亮横幅，标注文字说明 AI 在此帧的优势（如"AI 识别到悬空障碍物，传统算法误报为墙体"），持续显示 3 帧后自动消失。

    └── 检查通过 → 加载 DLL，完成 ctypes 函数绑定
            │
            ▼
        扫描 TestData/ 目录，构建会话索引
            │
            ▼

            │
            ▼

### 7.2 单会话回放流程

```
用户在W1选择会话并点击"加载"
    │
    ▼
数据加载器解析 esi_frames.bin，建立帧索引
    │
    ▼
加载 session_meta.json，填充W1元数据显示区
    │

    │
    ▼（用户点击播放）
回放主循环（10Hz定时器驱动）：
    ├── 从数据加载器获取当前帧数据
    ├── 传统引擎处理当前帧 → 结果融合模块 → 通知传统显示面板刷新
    ├── AI 引擎处理当前帧 → 结果融合模块 → 通知 AI 显示面板刷新
    ├── W1 更新帧元数据显示
    ├── 若启用GT评估：更新两侧 W6 统计数据
    └── 推进帧指针

    │
    ▼

```

## 8. 模块间接口设计

### 8.1 数据帧对象规格

数据管理模块向算法引擎提供的数据帧对象包含以下字段：

| 字段名 | 数据类型 | 维度 | 说明 |
|--------|---------|------|------|
| `frame_id` | int | 标量 | 帧在会话中的序号（0-based） |
| `timestamp_ms` | float | 标量 | 帧时间戳（ms） |
| `edi_distance` | float32数组 | [12] | 12路 EDI 距离（m） |
| `edi_amplitude` | float32数组 | [12] | 12路 EDI 幅度（归一化） |
| `edi_confidence` | uint8数组 | [12] | 12路 EDI 置信度（0~100） |
| `edi_echo_type` | uint8数组 | [12] | 12路回波类型枚举（0~3） |
| `envelopes` | float32数组 | [12, 256] | 12路归一化包络序列 |
| `vehicle_speed` | float | 标量 | 车速（km/h） |
| `steering_angle` | float | 标量 | 转向角（°，左负右正） |
| `gear` | str | 标量 | 档位（P/R/N/D） |

### 8.2 算法引擎输出结果规格

算法引擎处理完一帧后返回的结构化结果：

| 字段名 | 数据类型 | 维度 | 说明 |
|--------|---------|------|------|
| `frame_id` | int | 标量 | 对应帧编号 |
| `valid_flags` | float32数组 | [12] | M1：各路有效性概率（0~1） |
| `class_probs` | float32数组 | [12, 9] | 12路×9类概率（统一9类，传统算法未覆盖的类别概率为0） |
| `class_ids` | uint8数组 | [12] | 主分类ID（argmax结果，取值范围 0~8，遵循统一障碍物类别规范）|
| `sc_class_probs` | float32数组 | [12, 9] | M2-SA 启用时：M2-SC 骨干精化前的 9类概率（AI专有；SA 禁用时与 class_probs 相同） |
| `sc_class_ids` | uint8数组 | [12] | M2-SA 启用时：M2-SC 骨干精化前的主分类ID（AI专有；SA 禁用时与 class_ids 相同） |
| `material_hardness` | float32数组 | [12] | M2：材质硬度评分（0~1，AI专有） |
| `suspension_height_m` | float32数组 | [12] | M2：悬空净高估算（m，非悬空类填-1，AI专有） |
| `inference_time_ms` | float | 标量 | 本帧推理耗时（ms，用于性能监控） |
| `engine_type` | str | 标量 | 来源标识（"AI"或"Traditional"） |

### 8.3 显示模块刷新接口

结果融合模块通过**观察者模式（Observer Pattern）**通知各显示窗口刷新，各窗口订阅自己感兴趣的数据字段：

| 窗口 | 订阅的数据字段 |
|------|-------------|
| W2 包络波形视图 | `envelopes`、`valid_flags` |
| W3 分类视图 | `class_probs`、`class_ids`、`sc_class_probs`、`sc_class_ids`、`valid_flags`、`material_hardness`、`suspension_height_m` |
| W4 OGM | `class_ids`、`valid_flags`、`edi_distance`、`vehicle_speed`、`steering_angle` |
| W6 统计面板 | 全部字段（用于 GT 比对） |

## 11. 硬件在环（HIL）扩展接口

本章描述仿真软件在具备实体传感器节点但无完整量产ECU的实验室环境下，通过 PC 驱动真实硬件进行闭环测试的扩展能力。HIL 模式属于**可选扩展**，不影响第3~10章描述的纯离线仿真功能。

### 11.1 HIL 测试场景与适用条件

| 适用场景 | 硬件条件 | 相对于离线仿真的优势 |
|---------|---------|-------------------|
| 传感器特性验证 | PC + USB-ESI桥接器 + DJ628.27实体节点×N | 获取真实传感器原始信号，排除数据采集记录误差 |
| 算法实时性验证 | 同上 | 在真实10Hz控制周期约束下验证算法响应延迟 |
| 传感器老化/环境适配 | 同上 + 温控箱（可选） | 直接采集高/低温下的真实包络数据 |
| ECU集成预验证 | PC + USB-ESI + 目标ECU开发板 | 在量产ECU上验证SDK推理与PC端DLL推理的数值一致性 |

### 11.2 USB-ESI 桥接器接入方案

**桥接器功能要求**：
- 支持 ESI（基于DSI3）协议的完整物理层和数据链路层
- 能够正确处理 ESI 的同步触发时序（Chirp触发 → 回波接收窗口 → EDI上报 → 包络传输）
- 以 USB HID 或 USB CDC 设备形式接入 PC，提供开放的 PC 端驱动 API
- 支持最多16路 DJ628.27 节点同时在线

**软件切换方式**：在 `config.yaml` 中将 `engine.data_source` 从 `"file"` 改为 `"hil_usb"`，并填写桥接器的 USB 设备路径（`hil.device_path`），重启软件后数据管理模块自动切换为实时 USB 接收模式。上层算法引擎、显示模块、对比模块完全不需要修改，与离线仿真模式使用完全相同的处理链路。

**数据流向（HIL模式）**：
```
DJ628.27 实体节点（最多16路）
        │ ESI总线
        ▼
USB-ESI 桥接器
        │ USB CDC/HID
        ▼
PC 端桥接器驱动（提供与文件回放相同的帧对象接口）
        │ 内部帧对象队列
        ▼
算法引擎 → 显示模块（与离线模式完全相同）
```

### 11.3 仿真数据注入 ECU 验证接口

当需要验证目标 ECU 完整软件栈（含 AUTOSAR 层及应用层，超出 SDK 范围）时，可通过以下方式将仿真软件的历史数据注入 ECU 进行端到端验证：

#### 11.3.1 SWD 内存注入方式

通过 J-Link 或 ST-Link 的脚本接口（JLinkScript / GDB server），将历史数据文件中的帧数据逐帧写入目标 ECU 内存中 SDK 的输入缓冲区地址，触发 SDK 运行推理，再通过 SWD 读取输出缓冲区内容，与 PC 端 DLL 推理的期望输出对比。

**验收标准**（与 AK2 AI模型开发说明 §7.4 一致）：
- 分类标签一致率 ≥ 98%（100条样本）
- Softmax 输出最大差值 ≤ 0.02

#### 11.3.2 CAN 总线旁路监听方式

通过 USB-CAN 适配器监听 ECU 向整车 CAN 总线发送的感知输出报文（障碍物距离报文、碰撞确认报文），将监听到的 CAN 报文按 DBC 定义解析后，与仿真软件在 PC 端生成的期望输出对比，作为系统级集成测试的自动化验证手段，无需 SWD 接入。

### 11.4 HIL 相关配置项（config.yaml 补充）

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `hil.enabled` | `false` | 是否启用 HIL 模式 |
| `hil.device_path` | `"COM3"` | USB-ESI 桥接器的串口路径（Windows）或设备文件（Linux） |
| `hil.sensor_count` | `12` | 在线传感器路数 |
| `hil.frame_timeout_ms` | `150` | 等待桥接器返回完整帧的超时时间（ms），超时后用零填充缺失路数据 |
| `hil.record_to_testdata` | `false` | 是否将 HIL 实时数据同步录制到 `TestData/` 目录（用于后续离线回放） |
