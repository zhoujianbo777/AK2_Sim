## 5. AI 推理引擎层

### 5.1 推理引擎架构与部署方式

#### 5.1.1 三层分层推理架构概述

AI 推理引擎层是本 SDK 的核心计算层，其内部采用三层分层架构设计，将感知推理任务按照实时性要求、计算量和触发条件的不同，分配到三个独立运行的 AI 模块中。三层架构的核心设计哲学是"快速路径绝对优先"：实时性最高的第1层在每个控制周期必须完成并向 APA 控制环输出结果，而第2层和第3层以异步方式补充更深层次的感知理解，不影响实时控制路径的时间确定性。

三层模块的分工对比如下：

| 层次 | 模块名称 | 触发方式 | 延迟目标 | 主要输入 | 主要输出 |
|------|---------|---------|---------|---------|---------|
| 第1层 | 快速特征融合（MLP+规则引擎） | 每控制周期必执行 | <2ms | 12路EDI特征向量（240维） | 障碍物有效性标志、加权距离、碰撞初步标志 |
| 第2层-A | 障碍物深度分类（1D-CNN） | 按需触发（由传输策略管理模块触发） | <15ms（含包络传输） | 单路归一化包络（1×256） | **9类**概率分布、材质软硬度评分、悬空高度估算（V1.2） |
| 第3层 | 场景理解（2D-CNN+贝叶斯OGM+卡尔曼） | 每2~5个控制周期执行一次 | <30ms | 多路包络矩阵（12×256）+车辆运动状态 | OGM栅格地图、车位识别结果、障碍物轨迹 |

三层之间的数据依赖关系为：第1层的输出（有效性标志和碰撞初步标志）是触发第2层的判定条件；第1层和第2层的综合输出共同构成第3层的输入。三层的输出均向融合决策层（第6章）提供数据，由融合决策层汇聚后对外输出统一的感知结果。

#### 5.1.2 推理引擎的内存布局

AI 推理引擎层占用的内存由三部分组成：模型权重存储区、推理工作内存区（激活值缓冲）和输入/输出数据缓冲区。

**模型权重存储区**：量化模型的权重文件存储在 Flash/ROM 中（只读），在 SDK 初始化时根据配置可选择将权重复制到 SRAM 以提升推理速度（代价是占用更多 SRAM），或直接从 Flash 中逐层读取（SRAM 占用更少，但推理速度略慢）。各模型 INT8 量化后的权重大小参见第8章资源规格。

**推理工作内存区**：每个模型在推理过程中需要临时存储各层的激活值（中间计算结果），其大小取决于网络最大层激活值的尺寸。三个模型的工作内存可以复用同一块 SRAM 区域（因为三个模型不会同时运行），需在 SDK 初始化时分配足够的共享工作内存缓冲区，大小为三个模型工作内存需求的最大值。

**输入/输出数据缓冲区**：存放模型的量化输入（INT8格式）和推理输出（INT8格式，反量化后为float32），各模型的输入输出格式在5.2~5.4节详细定义。

#### 5.1.3 量化推理内核的集成方式

SDK 的 AI 推理内核通过工具链生成的标准 C 代码方式集成，不依赖第三方推理框架的动态链接库，确保可在任何嵌入式编译环境下编译。三种推理工具链的支持情况如下：

| 工具链 | 典型支持目标芯片 | 支持算子范围 | 量化方式 |
|--------|--------------|-----------|---------|
| STM32Cube.AI | STM32/SPC5系列（Cortex-M/R） | Conv1D/2D、FC、池化、激活函数、Softmax | PTQ（训练后量化）INT8 |
| NXP eIQ Toolkit | S32系列（Cortex-A/R） | 同上，另支持部分LSTM算子 | PTQ INT8 |
| Apache TVM | 通用（需配置目标架构） | 最广泛，支持所有主流算子 | PTQ/QAT均支持 |

各工具链均接受 ONNX 格式的模型输入（从 PyTorch/TensorFlow 训练的模型导出），输出为可直接编译到 ECU 的 C 源文件和头文件。SDK 通过统一的推理调用接口屏蔽不同工具链实现的差异，上层模块只需调用标准推理接口，无需了解底层工具链的具体实现。

### 5.2 第1层：快速特征融合模块（MLP + 规则引擎）

#### 5.2.1 输入特征向量定义（12路×20维，共240维）

第1层 MLP 的输入是一个240维的浮点特征向量，由12路传感器的 EDI 数据拼接而成，每路提供20个特征维度。12路的排列顺序与传感器在车身上的空间位置对应（从车身左侧到右侧、从前到后），这一固定排列顺序在 SDK 初始化时通过节点地址映射表确定，训练数据中必须使用相同的排列顺序。

**故障节点的特征处理**：当某路传感器处于故障或离线状态时（节点有效性标志=0.0），该路的其他所有特征维度全部填充为固定值0.0（距离相关维度）或特定标记值，并在特征向量元数据中标注该路为无效路。MLP 模型在训练时需包含含有故障路（特定模式全零段）的样本，确保模型对传感器缺失场景保持鲁棒性。

#### 5.2.2 MLP 模型接口规范

**网络结构**：第1层 MLP 采用3层全连接网络，结构为 240→128→64→输出，激活函数为 ReLU（输入层到第2层）和 Sigmoid/Softmax（输出层），中间层使用 Dropout（训练时比率0.2，推理时关闭）和 Batch Normalization（推理时使用训练后的均值/方差常数，无额外计算开销）。

模型规格汇总：

| 项目 | 规格 |
|------|------|
| 输入维度 | 216（12路×18维） |
| 隐藏层1 | 128个神经元，ReLU激活 |
| 隐藏层2 | 64个神经元，ReLU激活 |
| 总参数量 | 约4万（float32约160KB，INT8量化后约40KB） |
| 推理计算量 | 约0.5MFLOPS |
| 推理延迟（典型） | <1ms（Cortex-R52@400MHz） |
| 量化格式 | INT8，推理后反量化 |

**输入数据缩放**：240维特征向量在送入模型推理前，需按照模型训练时使用的归一化参数进行标准化，各维度独立进行（减去训练集均值，除以训练集标准差）。归一化参数以常数数组形式存储在 SDK 中，随模型一起分发，确保训练和推理使用相同的数据分布。

**输出后处理**：MLP 输出为各路障碍物的有效性得分（Sigmoid激活，范围[0,1]），通过可配置阈值（默认0.5）转换为二值有效性标志。实际部署中，阈值可根据对虚警率和漏检率的偏好进行调整（提高阈值→降低虚警率；降低阈值→降低漏检率），不需要重新训练模型。

#### 5.2.3 规则引擎配置与触发条件

规则引擎与 MLP 并行运行，作为 MLP 的补充保障层，用于处理 MLP 统计学习能力不擅长的极端确定性场景。规则引擎的判断结果优先级高于 MLP 输出——当规则引擎判定为"强确认"时，直接覆盖 MLP 输出；当规则引擎判定为"强否定"时，同样直接覆盖 MLP 输出；其余情况两者取MLP输出。

规则引擎包含以下强制判定规则：

| 规则名称 | 触发条件 | 输出结果 |
|---------|---------|---------|
| 近场强制有效 | 任意路距离<0.20m（NFD近场）且置信度≥128 | 该路强制标记为有效，且障碍物类别暂定为"需深度分类" |
| 超近距离碰撞预警 | 距离<0.10m且置信度≥200 | 该路强制标记有效，并触发APA紧急停车请求（不等待第2层分类） |
| 全路无效判断 | 12路EDI均无有效回波（回波数量=0）连续3帧 | 输出"视野全空"标志，第1层输出有效距离置为最大值 |
| 节点离线无效 | 节点有效性标志=0 | 该路强制标记无效，覆盖MLP可能的误判 |

规则引擎的所有阈值参数均以可配置形式存储，支持在 SDK 初始化时通过配置结构体覆盖默认值，以适应不同车型（如低底盘车型的超近距离参数调整）或应用场景（泊车模式 vs 低速驾驶模式）的差异化需求。

#### 5.2.4 输出：障碍物有效性标志 / 置信度加权距离 / 碰撞初步标志

第1层模块的输出数据结构定义如下，该结构体每个控制周期更新一次，更新延迟<2ms（自 EDI 帧全部入队起计时）：

**12路障碍物有效性标志数组**（12个 uint8，1字节/路）：每路取值 0（无效/无障碍物）或 1（有效/有障碍物），由 MLP 输出阈值化或规则引擎强制覆盖得出。APA 应用层可直接读取此数组作为最快速的障碍物存在性判断，无需等待第2层的语义分类。

**12路置信度加权距离数组**（12个 float32，4字节/路）：每路的最近障碍物距离值经过 EDI 置信度加权融合后的输出，单位米。计算方式为：将本路当前帧的 EDI 置信度（归一化至[0,1]）作为权重，对本帧距离值与上一帧距离值进行加权滑动平均，得到平滑后的距离估计。无有效障碍物时输出最大量程值（6.5m）。此输出适合直接送入 APA 路径规划模块作为障碍物距离输入。

**碰撞初步标志**（uint8，1字节）：当规则引擎超近距离碰撞预警规则触发（任意路距离<0.10m 且置信度≥200）时，该标志置1。正常行驶无碰撞时恒为0。

**系统健康状态字**（uint16，2字节）：包含当前控制周期内的各类异常状态位，包括：有效节点数量（4bit，当前在线传感器数量）、EDI 解析错误计数（4bit）、规则引擎强制覆盖次数（4bit，用于统计规则引擎介入频率，辅助调试）、第1层 MLP 是否超时（1bit）、保留位（3bit）。APA 应用层和上位机调试工具可通过此字段了解当前系统工作质量。

### 5.3 第2层：深度分类模块（按需触发）

#### 5.3.1 子模块A：1D-CNN 障碍物分类器

##### 5.3.1.1 网络结构与参数规格

1D-CNN 障碍物分类器对单路传感器的归一化包络序列进行特征提取和分类，是实现障碍物类型感知的核心模型。网络采用三层卷积 + 全局均值池化 + 两层全连接的标准轻量架构，在提供充足分类精度的同时，将参数量和计算量严格控制在车规 ECU 可承载的范围内。

**完整网络层次规格**：

| 层次 | 层类型 | 配置参数 | 输入维度 | 输出维度 | 参数量估算 |
|------|------|---------|---------|---------|---------|
| 输入层 | 数据输入 | — | 1×256 | 1×256 | 0 |
| 第1卷积层 | Conv1D + BN + ReLU | 16通道，核长7，步长1，padding=same | 1×256 | 16×256 | 16×(1×7+1) = 128 |
| 第1最大池化层 | MaxPool1D | 池化窗口2，步长2 | 16×256 | 16×128 | 0 |
| 第2卷积层 | Conv1D + BN + ReLU | 32通道，核长5，步长1，padding=same | 16×128 | 32×128 | 32×(16×5+1) = 2592 |
| 第2最大池化层 | MaxPool1D | 池化窗口2，步长2 | 32×128 | 32×64 | 0 |
| 第3卷积层 | Conv1D + BN + ReLU | 64通道，核长3，步长1，padding=same | 32×64 | 64×64 | 64×(32×3+1) = 6208 |
| 第3最大池化层 | MaxPool1D | 池化窗口2，步长2 | 64×64 | 64×32 | 0 |
| 全局均值池化层 | GlobalAvgPool1D | — | 64×32 | 64 | 0 |
| 全连接层1 | FC + ReLU + Dropout(0.3) | 64→32 | 64 | 32 | 64×32+32 = 2080 |
| 全连接层2（输出层） | FC + Softmax | **32→9**（V1.2扩展） | 32 | **9** | 32×9+9 = 297 |
| **合计** | — | — | — | **9类概率** | **约11K参数** |

> **说明**：Batch Normalization 层（BN）在推理时使用训练阶段统计得到的均值和方差常数，不增加推理计算量，仅增加少量常数存储开销（每个BN层4个参数向量）。上表参数量估算已包含BN层参数，卷积层参数量 = 输出通道数 × (输入通道数 × 核长 + 1)（含偏置）。实际部署中工具链可能将BN层融合进卷积层，进一步减少推理运算量。

**完整参数量汇总**：基础参数约1.1万个，考虑BN层参数后约6万个（与可行性分析文档中的估算一致），float32约240KB，INT8量化后约60KB。推理时 MACs 约 1MFLOPS。

**输入数据规格**：归一化包络序列 1×256，float32，取值范围[0.0, 1.0]，由第4章包络预处理模块提供。可选地，在序列末尾追加余振频率偏差和余振持续时间两个辅助标量特征（对应第4.1.4节所述的辅助特征），此时输入维度变为 1×258。是否启用辅助特征为可配置项，启用时需与模型训练时的输入格式一致。

##### 5.3.1.2 推理接口定义

1D-CNN 分类器的推理调用遵循以下规格，由第2层 AI 调度逻辑在策略管理模块触发包络传输并完成预处理后调用：

**调用时机**：在包络预处理模块完成256点对齐并将归一化包络写入输入缓冲区后，由调度管理模块调度1D-CNN推理任务运行。对于同一控制周期内触发的多路包络传输请求，分类器对每路包络独立推理，推理顺序按置信度从高到低执行（优先分类更可靠的目标）。

**推理时长控制**：单路推理延迟目标<5ms（不含包络传输时间）。若单路推理超过8ms，该路推理被中止，输出标记为"推理超时"，该路障碍物分类结果在本控制周期保留上一次有效分类结果。

**并发控制**：第2层-A 的推理任务不得与第1层的 MLP 推理同时运行（第1层完成后方可启动第2层-A，确保第1层的实时性不受影响）。

**推理结果的有效期管理**：每路障碍物的分类结果具有有效期属性（默认5秒=50个控制周期）。当某路超过有效期内未重新触发分类推理时，该路的分类结果标记为"过期"，融合决策层对过期结果的置信度自动降低50%。有效期可通过传输策略管理模块的"定期刷新触发"规则（见第4.4.3节）来保证分类结果的时效性。

##### 5.3.1.3 输出：9类障碍物概率分布 + 材质软硬度评分 + 悬空高度估算（V1.2）

> **V1.2 变更说明**：在原有6类基础上新增类别6（悬空/高位障碍物）、类别7（路沿/低矮障碍物）、类别8（湿滑/镜面地面），对标 BOSCH USS7 场景 F/G/H/I。训练数据集须同步扩充三类新场景的标注样本（建议每类不少于500条），量化校准集须覆盖全部9类。

**9类障碍物概率分布**：Softmax 输出层输出9个概率值，各概率值之和为1.0：

| 类别索引 | 障碍物类型 | 典型场景 | APA应用意义 |
|---------|---------|---------|-----------|
| 0 | 墙体/混凝土柱/硬质刚性障碍物 | 停车场墙体、立柱、路牙 | 安全裕度最大（20cm以上），严格规避 |
| 1 | 车辆（静止或慢速运动） | 旁边停放的车辆 | 安全裕度大（15cm），规划绕行路径 |
| 2 | 行人 | 行人穿越、旁观者 | 安全裕度最大（30cm以上），立即停车等待 |
| 3 | 软物/植被/轮挡/锥桶 | 草丛、停车轮挡、橡胶锥桶 | 安全裕度小（5cm），低速可轻微接触 |
| 4 | 空旷/无障碍物/噪声 | 开阔区域、误报回波 | 不作为障碍物处理 |
| 5 | 地面杂波/路面反射 | 路面标线、坡道边缘（无湿滑） | 忽略（非悬空障碍物） |
| **6** | **悬空/高位障碍物** | 地库限高横梁、挡车臂（水平障碍）、卷帘门底边 | **V1.2新增**：结合悬空高度估算输出判断是否可下穿；净高不足则规避 |
| **7** | **路沿/低矮障碍物** | 路牙（高5~15cm）、坡道金属边缘、地面水泥台阶 | **V1.2新增**：横向安全裕度缩至3cm以内，APA需评估跨越可行性；配合5.4.6路沿探测接口使用 |
| **8** | **湿滑/镜面地面** | 积水地面、光滑石材地面、覆冰路面 | **V1.2新增**：本身不作为障碍物，但触发"低μ路面"标志，APA控制环需降低最大横向加速度限值 |

APA 应用层在使用分类结果时，应以各类别中概率最高者作为主类别，同时参考主类别概率的绝对值（低于0.5时表示分类置信度不足，应保守处理）。类别8（湿滑地面）在主类别概率>0.6时，无论当前是否存在障碍物，均应向 APA 速度控制器输出"低μ路面"标志，触发最大横向加速度限值从默认值降低30%。

**材质软硬度评分**（附加输出，float32，范围[0.0, 1.0]）：由全连接层1的激活值通过附加一个回归头计算得到，0.0代表最软（垂草、薄布），1.0代表最硬（混凝土、钢铁）。软硬度评分与类别概率共同用于 APA 路径规划中的安全裕度差异化。

**悬空高度估算**（V1.2新增，float32，范围[0.0, 3.5]，单位m）：仅当主类别=6时有效，估算障碍物底面到地面的净高。原理：分析包络序列中地面回波（时延较长）与高位障碍物回波（时延较短）在时间轴上的双峰间距，结合传感器安装高度（SDK 初始化时配置），推算净空高度。精度目标±10cm。非类别6时该字段填0.0，APA 应用层应忽略。净高输出将作为 5.4.5 节负向空间推断模块的辅助融合输入。

### 5.4 第3层：场景理解模块（周期运行）

#### 5.4.1 贝叶斯 OGM 占用栅格地图更新接口（5cm 分辨率）

##### 5.4.1.1 OGM 地图结构定义

OGM（Occupancy Grid Map，占据栅格地图）将车辆周围的二维感知区域划分为等间距的正方形栅格，每个栅格存储该位置被障碍物占据的概率值。地图参数定义如下：

| 参数 | 数值 | 说明 |
|------|------|------|
| 栅格尺寸 | 5cm × 5cm | 包络序列精度支撑的最高分辨率 |
| 地图范围 | 前向8m × 侧向6m（单侧3m） | 覆盖典型泊车操作范围 |
| 地图尺寸 | 160×120 栅格 = 19,200 个栅格 | 每个栅格1字节，地图总大小约19KB |
| 坐标原点 | 车辆前轴中心 | 随车辆运动实时更新（滑动窗口） |
| 概率表示 | uint8（0~255），0=空闲，255=被占据 | 贝叶斯更新后的对数几率比转换 |
| 初始值 | 128（对应先验概率0.5，不确定） | 未观测栅格保持不确定状态 |

地图的概率更新采用贝叶斯对数几率比（Log-Odds）形式，避免直接对概率进行乘法运算（数值稳定性更好）。每次新的传感器观测到来时，对对应栅格的对数几率比值进行加减更新，然后转换回概率值存储。

##### 5.4.1.2 OGM 更新触发与流程

**触发时机**：第3层场景理解模块每2~5个控制周期执行一次完整的 OGM 更新。触发间隔可根据 ECU 当前算力余量动态调整（算力充裕时每2周期更新一次，算力紧张时降至5周期更新一次），由调度管理模块（第7章）控制。

**OGM 更新流程**：

第一步，地图坐标变换。根据车辆运动状态（车速、转向角、里程计，来自 CAN 总线输入），将上一帧 OGM 地图中的所有栅格坐标变换到当前帧的车辆坐标系中（刚体坐标变换，对应车辆的位移和转向），实现地图随车辆运动的"滑动跟随"。坐标变换采用双线性插值，计算量约0.5MFLOPS。

第二步，包络投影。将本控制周期内收到的多路包络序列（若采用策略B全量传输，则为12×256的包络矩阵）投影到 OGM 栅格坐标系中：以传感器安装位置为原点，以包络序列的时间轴（对应距离轴）为方向，将包络幅值分布投影到径向栅格上，形成"回波能量栅格图"。幅值较高的栅格（超过 CFAR 后处理确定的障碍物检测阈值）被标记为有障碍物候选区域。

第三步，贝叶斯更新。对每个被包络投影标记为"有障碍物"的栅格，将其对数几率比增加固定值（命中增量，默认+1.5，对应概率从0.5更新到约0.82）；对包络覆盖区域内未被标记（回波稀疏区域，对应声波穿透、无障碍物）的栅格，将其对数几率比减少固定值（空闲增量，默认-0.5，对应轻微降低占据概率）；未被本次包络覆盖到的栅格（超出传感器探测范围，或本次未触发包络传输的路覆盖区域）不更新，保留历史概率。

第四步，老化衰减。对超过10个控制周期（约1秒）未被更新的栅格，对数几率比向初始值（128/不确定）方向衰减一小步（衰减因子可配置，默认每10周期向不确定方向衰减5%），使地图能够自动"遗忘"消失的静态障碍物（例如停车后移走的锥桶），避免地图长时间积累过时的障碍物信息。

#### 5.4.2 车位角点时序检测接口

车位边界检测基于车辆匀速行驶过程中，侧向传感器包络序列的时序变化特征来识别车位的开口和闭合位置，进而推断车位的尺寸和方向。

**检测原理**：当车辆匀速驶过一排停放的车辆时，侧向传感器（前左角/左侧/后左角）的包络序列会呈现典型的时序变化模式：停放车辆存在时，回波峰值距离较近（约0.5~1.5m）且包络幅值大；车位空缺处，回波距离突然变远（>2m）或消失；下一辆停放车辆出现时，回波距离恢复至近距。这种"近-远-近"的距离跳变序列构成车位开口的特征指纹。

**时序缓冲区管理**：SDK 维护约100帧（10秒）的侧向传感器 EDI 历史数据缓冲区（按固定环形缓冲区实现，每帧存储3路侧向传感器的距离和置信度，每帧约18字节，100帧约1.8KB），同时记录每帧对应的车辆里程计读数，用于将时序帧序号转换为车辆行驶距离。

**车位角点检测算法**：在每次第3层更新触发时，对历史距离序列运行滑动窗口变化点检测（Change Point Detection），识别距离序列中的跳变位置（对应车位边界）。具体判据为：若某一帧的距离值与其前后各5帧的均值差超过0.8m，则该帧标记为一个变化点（车位边界候选）；若变化点之间的行驶距离在2.2~6.0m范围内（对应常见车位长度），则该区间识别为有效车位。

**车位尺寸估算**：车位长度由两个变化点之间对应的里程计差值直接计算；车位深度（垂直于行驶方向的深度）由变化点处的包络幅值宽度（回波拖尾长度对应目标深度）推算，估算精度约±20cm。

**输出数据结构**：车位识别结果包含：车位类型（平行/垂直/斜列，枚举）、车位开口角点坐标（车身坐标系，2个角点×2维坐标，float32）、车位长度估算（float32，米）、车位深度估算（float32，米）、识别置信度（uint8，0~255）。

#### 5.4.3 障碍物轨迹卡尔曼滤波接口

对于动态障碍物（行人、缓慢移动的车辆），SDK 使用卡尔曼滤波对其运动轨迹进行跟踪和预测，提供比单帧 EDI 距离更平滑、更可靠的障碍物位置估计，并支持对障碍物未来位置的短期预测（预测1~3个控制周期，即100~300ms内的位置）。

**状态向量定义**：每个被跟踪目标的卡尔曼滤波状态向量包含4个维度：纵向位置（x，车身坐标系，米）、横向位置（y，车身坐标系，米）、纵向速度（vx，米/秒）、横向速度（vy，米/秒）。状态协方差矩阵为4×4对称矩阵。

**观测量**：卡尔曼滤波的观测输入为第1层 MLP 输出的置信度加权距离，结合传感器安装位置和传感器指向角，将径向距离转换为 x-y 坐标作为观测向量。

**目标关联**：当有新的观测数据到来时，SDK 使用最近邻（Nearest Neighbor）算法将新观测与已有跟踪目标进行关联（匈牙利算法，时间复杂度O(n³)，n为当前跟踪目标数量，APA 场景下通常n<10，计算量可忽略）。关联成功则更新对应目标的状态；关联失败（无匹配目标）则新建跟踪目标；长时间（默认5个控制周期）未收到观测更新的目标标记为"消失"并从跟踪列表中删除。

**输出预测**：对每个活跃跟踪目标，卡尔曼滤波提供当前位置估计（经过测量噪声滤波后的平滑位置）和未来3帧的预测位置（用于 APA 路径规划的前瞻性避障）。

#### 5.4.4 综合输出：精细障碍物地图 / 车位识别结果 / 碰撞事件

第3层场景理解模块的综合输出在每次触发运行后更新，向融合决策层（第6章）提供以下三类数据：

**精细障碍物地图（OGM）**：160×120 uint8 矩阵，代表5cm×5cm分辨率的障碍物概率地图，以指针形式（不拷贝）传递给融合决策层和上层 APA 路径规划模块。地图数据在下一次 OGM 更新完成前保持稳定有效。APA 路径规划模块读取 OGM 用于生成避障约束，将概率>200（对应占据概率>78%）的栅格视为障碍物，200~100之间的栅格视为不确定区域（需保守处理），100以下视为空闲区域。

**车位识别结果列表**：当前检测到的所有有效车位的列表（最多8个），每个车位包含第5.4.2节所述的完整车位数据结构。若无有效车位，列表为空。列表在每次第3层触发时更新。

**动态障碍物跟踪列表**：当前所有活跃跟踪目标的状态列表（最多16个），每个目标包含当前估计位置、速度、未来3帧预测位置。静态障碍物（连续10帧速度估计<0.1m/s）自动标记为静态，不提供速度预测信息。

### 5.5 模型量化与部署规范

#### 5.5.1 INT8 量化规范（精度损失 <2%）

SDK 中所有 AI 模型均采用 INT8（8位整数）量化后部署，将 float32 的权重和激活值量化为 int8，推理速度提升约2~4倍，模型内存占用减少约75%，精度损失目标控制在2%以内（以在验证集上的分类准确率相对下降量衡量）。

**量化方案选择**：采用训练后量化（Post-Training Quantization, PTQ）方案，在训练完成后使用代表性校准数据集（从训练集中抽取约200~500个样本，覆盖全部6类障碍物场景）对量化参数（缩放因子和零点）进行标定。对于 MLP 类模型（参数量少、结构简单），PTQ 通常即可满足<2%精度损失要求；对于 2D-CNN（参数量大、对量化敏感），若 PTQ 导致精度下降超过2%，应升级为量化感知训练（Quantization-Aware Training, QAT），在训练阶段模拟量化误差，使模型在量化后仍保持高精度。

**量化精度验收基准**：量化后模型须在专用的量化验证数据集（与训练集和测试集均不重叠的独立数据集，至少1000条样本）上通过以下验收标准：

| 验收指标 | 验收标准 |
|---------|---------|
| 1D-CNN 障碍物分类准确率（量化后 vs float32） | 相对下降<2%（例如float32准确率88%，量化后≥86.2%） |
| MLP 碰撞分类准确率（量化后 vs float32） | 相对下降<2% |
| 2D-CNN 车位识别精度（角点误差） | 均方根误差增加<5cm |
| TDOA 定位精度 | 不受量化影响（纯数学计算，无量化误差） |
| 推理延迟（量化后 vs float32） | 提升≥1.5倍 |

**量化参数存储**：每个模型的量化参数（各层的缩放因子和零点值）以常数数组形式与模型权重一起存储在 SDK 的 Flash/ROM 区域，在 SDK 初始化时由推理内核自动加载，用户无需手动管理量化参数。

#### 5.5.2 推理工具链集成（STM32Cube.AI / NXP eIQ / Apache TVM）

**集成流程**：模型从训练框架（PyTorch/TensorFlow）导出为 ONNX 格式后，经推理工具链转换为 ECU 可执行的 C 代码，集成步骤如下：

第一步，ONNX 导出与验证。将训练好的 float32 模型导出为 ONNX 格式，使用 onnxruntime 在 PC 端验证 ONNX 模型的推理结果与原生框架一致（误差<1e-5），确认 ONNX 导出无精度损失。

第二步，工具链量化转换。将 ONNX 模型送入所选推理工具链，输入校准数据集进行 INT8 量化标定，生成量化后的中间格式（各工具链格式不同：STM32Cube.AI 输出 .h5 格式，NXP eIQ 输出 .tflite 格式，Apache TVM 输出编译后的二进制）。

第三步，C 代码生成。从量化中间格式生成标准 C 代码（包含模型权重数组的 .c 文件和推理函数声明的 .h 文件），加入 ECU 工程编译。生成的 C 代码须通过 MISRA-C 2012 静态分析检查（工具链通常提供 MISRA 兼容选项）。

第四步，ECU 端集成验证。将生成的推理 C 代码集成到 SDK 的 AI 推理模块中，使用 SWD 调试接口在目标 ECU 上运行推理，与 PC 端 ONNX 推理的结果进行逐帧对比，验证 ECU 端输出与 PC 端参考输出的最大偏差在允许范围内（对于 Softmax 输出的概率值，允许偏差<0.01）。

#### 5.5.3 模型文件管理与版本控制

**模型版本标识**：每个量化模型文件携带版本号（格式：主版本.次版本.修订号，如 1.2.0），版本号存储在模型文件头部的固定偏移位置，SDK 初始化时读取并记录到系统状态区，供诊断工具查询。主版本号变更表示网络结构发生变化（不向下兼容），次版本号变更表示训练数据或量化参数更新（结构兼容），修订号变更表示修复了量化 bug（透明升级）。

**多模型版本共存**：Flash 中为每个模型预留两个存储分区（主分区和备用分区），支持在不中断系统运行的情况下将新版本模型写入备用分区，然后通过配置切换在下次系统重启时激活新模型。若新模型在上线后的运行监控中出现性能异常（分类结果置信度持续低于阈值的比例超过10%），SDK 可自动回滚到备用分区中的上一个版本。

**模型完整性校验**：每次 SDK 初始化时，对所有模型权重文件执行 CRC32 完整性校验，校验值与模型文件中存储的预计算 CRC32 对比，不一致则报告"模型文件损坏"告警并阻止推理运行，防止因 Flash 位翻转导致模型权重损坏后产生危险的错误推理结果。

## 6. 融合算法模块

### 6.1 超声波通道处理流程

#### 6.1.1 12路1D-CNN并行推理管理

超声波通道的深度分类处理以12路传感器各自独立的1D-CNN推理为基础，但受限于 ESI 总线带宽和包络传输时序，12路并非严格同时运行推理，而是按照传输策略管理模块的调度顺序，以流水线方式依次完成各路的包络传输和推理。

**12路推理的流水线调度方式**：在一个控制周期（100ms）内，传输策略管理模块最多对4路传感器触发完整包络传输（见第4.4.3节策略C触发条件）。每路的完整处理时序为：包络上传命令发出→包络数据帧接收（约11ms）→预处理（约2ms）→1D-CNN推理（约5ms），单路合计约18ms。在100ms控制周期内串行处理4路所需时间约72ms，加上第1层MLP处理（<2ms）和其他系统开销，总时间在100ms以内，满足实时性要求。

若同一周期内有超过4路同时触发（如车辆第一次进入停车场，全部12路突然检测到障碍物），超出4路的请求按置信度排队，分散到后续控制周期内依次处理。对于排队中的传感器路，融合决策层使用该路上一次有效的分类结果（可能是上一周期或更早的结果），并自动标注该结果为"待刷新"状态。

**各路推理结果的生命周期管理**：为确保全部12路始终有可用的分类结果，SDK维护一个12路分类结果缓存表，结构如下：

| 字段 | 类型 | 含义 |
|------|------|------|
| 类别概率数组 | float32[9] | **9类**障碍物概率，由最近一次CNN推理输出（V1.2扩展） |
| 材质软硬度评分 | float32 | 取值[0,1] |
| 结果时间戳 | uint32 | 本次CNN推理完成时的系统时钟值（ms） |
| 有效期状态 | uint8 | 0=新鲜（<2s），1=有效（2~5s），2=过期（>5s），3=从未推理 |
| 结果版本号 | uint16 | 每次推理后递增，用于上层检测更新 |

分类结果缓存表在系统初始化时全部设置为"从未推理"状态，类别概率填充为均匀分布（每类1/9）。上层融合决策模块根据有效期状态对分类结果施加置信度折扣（新鲜=无折扣，有效=70%权重，过期=30%权重，从未推理=10%权重），确保过期数据不会对决策产生过度影响。

#### 6.1.2 EDI置信度加权概率矩阵生成

超声波通道的最终输出不是单纯的CNN分类概率，而是将CNN语义分类概率与EDI实时置信度进行加权融合，生成一个12×6的**置信度加权概率矩阵**，以反映每路当前测量状态下各类别的综合可信度。

**加权融合计算方式**：

对于第 i 路传感器（i=0~11），其融合后的6维类别概率向量 P_fused[i] 由以下三个因素共同决定：

第一因素是CNN分类概率向量 P_cnn[i]（由分类结果缓存表提供），这是模型根据历史或当前包络序列输出的类别先验；

第二因素是当前帧EDI置信度 c[i]（归一化至[0,1]，来自当前控制周期的EDI帧），反映当前测量时刻回波信号的质量；

第三因素是有效期权重 w_age[i]（根据分类结果的新鲜程度，取值0.1~1.0）。

融合公式为：P_fused[i] = w_age[i] × c[i] × P_cnn[i] + (1 - w_age[i] × c[i]) × P_uniform，其中 P_uniform 为均匀分布（每类1/9），代表"对该路无任何信息时的先验"。当EDI置信度极低（当前测量不可靠）或CNN结果过期时，融合结果自动向均匀分布靠拢，避免以不可靠的历史数据做出强烈的类型判断。

**12×6概率矩阵的用途**：融合后的12×6矩阵作为融合决策模块的核心输入，行方向对应12路传感器（空间维度），列方向对应6类障碍物（语义维度），矩阵中每个元素代表"在该路的当前状态下，该方向存在某类障碍物的综合可信度"。融合决策模块从这个矩阵中提取主类别、最高置信度、跨路一致性等指标，用于最终的障碍物决策输出。

### 6.2 融合决策接口

#### 6.2.1 融合判定规则配置

融合决策模块综合超声波通道（12×6概率矩阵）的输出，执行判定逻辑，得出最终的感知输出。融合判定规则以可配置参数结构体形式存储，支持在 SDK 初始化时按车型和应用场景进行定制。

融合判定规则的核心配置参数包括：

**超声波主类别置信度门限**（默0.5）：超声波通道12×6矩阵中，某路的最高类别概率超过此门限才被认为是有效的语义判断；低于此门限时，该路分类结果降级为“类型不明”。

**近距碰撞确认门限**（默0.10m）：某路距离小于此门限且置信度≥200时，触发碰撞初步标志。

**单模态降级模式**：当超声波通道故障路数超过4路时，融合决策降级为仅基于正常节点的超声波判断，并在健康状态字中记录故障路数。降级模式由节点状态机（见第3.5节）提供的故障信息自动触发，无需外部干预。

#### 6.2.2 碰撞确认逻辑（超声波类别概率）

最终碰撞确认输出由以下判定逻辑执行：

**碰撞确认条件**：某路超声波分类主类别为“墙体/车辆”类（类别0或1），且该路 EDI 置信度≥128，且 EDI 最近回波距离<0.10m（超近距离确认），则输出碰撞初步标志=1。

**近距预警条件**：某路 EDI 距离<0.20m 且置信度≥128，且超声波分类主类别为硬质目标类（类别0或1），则输出近距预警标志=1，暂不确认为碰撞。

**最终碰撞确认输出**：
- 碰撞确认条件成立：输出碰撞确认标志=1，触发 APA 紧急停车请求
- 仅近距预警条件成立：输出碰撞确认标志=0，近距预警标志=1，建议 APA 减速处理
- 两个条件均不成立：无碰撞，所有碰撞标志保持0

#### 6.2.3 综合置信度计算

综合置信度是对本次碰撞判定或障碍物感知结果可靠程度的量化估计，输出范围[0.0, 1.0]，计算方式如下：

**碰撞事件置信度**：

碰撞综合置信度 = 超声波主类别概率 × 超声波 EDI 置信度归一化値 × 连续帧数系数

其中：超声波主类别概率来自第2层-A输出（主类别的 Softmax 概率値）； EDI 置信度归一化値为该路置信度除以255；连续帧数系数对应单帧检测0.8，连续2帧0.9，连续3帧及以上1.0，反映连续检测越可靠。

**障碍物类别置信度**：

当前有效传感器路中，主类别概率最高路的概率值，乘以该路EDI置信度（归一化），再乘以分类结果新鲜度权重（新鲜=1.0，有效=0.8，过期=0.5），得到代表性障碍物类别置信度。若多路传感器对同一障碍物区域的分类结果高度一致（各路主类别相同且平均概率差异<0.1），置信度额外提升10%（多路一致性加成）。

### 6.3 两阶段碰撞确认机制

#### 6.3.1 阶段1：第1层快速判断接口（<2ms）

两阶段碰撞确认机制的第一阶段在每个控制周期内运行，由第1层 MLP 快速融合模块和规则引擎共同完成。

**第1层 MLP 快速判断**：12路 EDI 特征向量组装完成后，MLP 在<1ms 内输出12路障碍物有效性得分及置信度加权距离。MLP 得分高于阈値（默0.5）的路标记为有效障碍物路。

**规则引擎强制判定**：与 MLP 并行运行，处理短路/立即停车场景：任意路距离<0.10m 且置信度≥2000时，规则引擎立即置碰撞初步标志=1，并启动第二阶段。

**第一阶段时序**：EDI 帧全部到齐（自控制周期开始 T≈0ms起约35ms）→ MLP 推理（<1ms）→ 规则引擎判断（<0.5ms）→ 阶段1输出发布，全流程 T≈40ms 前完成。

**阶段1输出**：12路有效性标志数组、置信度加权距离数组、碰撞初步标志（尤其是超近距碰撞）。当碰撞初步标志=1时， APA 应用层可立即执行减速/停车预判逻辑（不等待第二阶段确认，先以预防性停车为主）；阶段1未输出碰撞类型和碰撞能量等级，这些精确信息由第二阶段提供。

#### 6.3.2 阶段2：ECU侧CNN精确分类确认接口（<15ms）

第二阶段在阶段1碰撞初步标志置1后由调度管理模块在下一个可用任务时间片中启动，执行第2层-A 1D-CNN对碰撞附近传感器路的包络分类，输出碰撞行为的精确语义确认。

**阶段2启动条件**：阶段1碰撞初步标志=1，且该路最近包络已成功接收并完成预处理。若包络未就绪，阶段2为该路发出包络上传命令，等包络传输完成后立即执行推理。

**阶段2时序**：包络传输（级11ms）→ 1D-CNN推理（<5ms）→ 融合决策输出更新，全流程在阶段1完成后<15ms内结束。

**与阶段1的协调机制**：APA 应用层在收到阶段1碰撞初步标志后，可选择立即执行“预防性停车”（以最大减速度停车，不等待阶段2确认），或执行“减速待命”（降低速度但不停车，等待阶段2确认后再决定是否停车）。两种策略的选择通过 SDK 配置参数控制：在车速较高（>5km/h）时推荐预防性停车，在车速极低（<1km/h）时推荐减速待命。

**阶段2输出的最终使用**：阶段2完成后，融合决策层输出最终碰撞事件结构体，包含第6.2.2节所述的完整碰撞确认信息。若阶段2确认为真实碰撞，且阶段1已触发预防性停车，则碰撞事件被记录到诊断日志（含时间戳、碰撞位置、碰撞目标类型），并通过 CAN 总线通知整车 BCM；若阶段2判定为误报，则通知 APA 解除预防性停车状态，恢复正常运行。

## 7. 系统时序与调度管理

### 7.1 控制周期时序规划

#### 7.1.1 100ms 控制周期总体时序

SDK 以100ms为基本控制周期，对应系统10Hz的感知刷新率。每个控制周期内，各任务按照固定的时序优先级顺序执行，确保实时性最强的第1层AI输出在周期的前段完成，为 APA 路径规划控制环留出充足的计算时间。

完整控制周期内的任务时序安排如下（以前6路传感器为例）：

| 时间节点 | 任务 | 执行位置 | 耗时估算 |
|---------|------|---------|---------|
| T=0ms | ECU下发Chirp测量触发指令（前6路广播） | ESI主机驱动 | <0.1ms |
| T=0~35ms | 前6路发波→声波飞行→回波接收（最远6.5m对应约38ms，近距约3ms） | 传感器节点侧 | 约3~38ms |
| T=35ms（典型） | 前6路EDI响应帧通过ESI总线上报完成（6帧×约0.5ms） | ESI主机驱动DMA接收 | <3ms |
| T=38ms | 第1层MLP快速融合运行（EDI特征向量240维） | AI推理引擎 | <2ms |
| T=40ms | 第1层输出：有效性标志+加权距离+碰撞初步标志，送APA控制环 | 融合决策层→APA | <0.1ms |
| T=40ms | 后6路Chirp触发（与前6路结果处理并行） | ESI主机驱动 | <0.1ms |
| T=40~50ms | 传输策略管理模块评估是否触发前6路包络传输 | 传输策略模块 | <1ms |
| T=50~61ms（按需） | 触发前6路中高优先级路的包络上传（单路约11ms） | ESI主机驱动 | 约11ms/路 |
| T=61~66ms（按需） | 第2层-A：1D-CNN障碍物分类（1路包络256点） | AI推理引擎 | <5ms |
| T=50~85ms | 后6路EDI+处理（与前6路包络传输并行） | 全栈 | <35ms |
| T=66ms | 第2层分类结果更新到分类结果缓存表，融合决策层更新12×6矩阵 | 融合决策层 | <1ms |
| T=70ms（每3~5周期执行一次） | 第3层：OGM更新+车位检测+卡尔曼滤波 | AI推理引擎 | <30ms |
| T=70~100ms | APA路径规划+轨迹跟踪控制计算 | APA应用层 | <30ms |
| T=100ms | 下一控制周期开始 | — | — |

**关键时序约束**：第1层输出最晚须在 T=45ms 前送达 APA 控制环，为路径规划留出至少55ms；第2层和第3层结果在周期内任意时刻更新均可，APA 应用层读取时使用最新可用结果（非阻塞读取）。

#### 7.1.2 前后向传感器组交替调度

前6路传感器和后6路传感器的测量周期相互交错，使整车的前向和后向感知均以10Hz运行，同时避免前后组同时发波导致的声学干扰（前后传感器在车身上距离较远，理论上干扰小，但软件上仍保持50ms错开触发作为预防）：

- **奇数控制周期**（第1、3、5…次）：T=0ms触发前6路，T=50ms触发后6路
- **偶数控制周期**（第2、4、6…次）：延续上一周期后6路的回波接收与处理；T=0ms同时触发新一轮前6路测量

两组数据在时间上的50ms固定偏差由第4章包络矩阵拼接中的时间戳标注机制记录，供第3层2D-CNN的位置编码层使用，确保场景理解模型感知到前后组数据的时间差。

#### 7.1.3 碰撞中断对正常调度的影响

碰撞中断帧到达时，正常调度流程按以下规则被打断：

**碰撞初步标志置1发生在 T=0~38ms（测量进行中）**：允许当前测量自然完成以获取超声波数据用于交叉验证。待本次测量的EDI帧上报完成后，优先执行第1层MLP，并弹出包络传输命令以开始第二阶段CNN分类确认。

**碰撞初步标志置1发生在 T=38~66ms（包络传输进行中）**：允许当前包络帧传输完成（剩余帧最多约11ms），传输完成后立即对碰撞附近路进行1D-CNN确认推理。已完成的1D-CNN推理结果保留在缓存表中，不需要重新推理。

**碰撞初步标志置1发生在 T=66~100ms（APA计算阶段）**：立即弹出包络传输命令并启动CNN确认推理，不影响当前正在执行的APA路径规划。CNN碰撞确认结果在本周期结束前写入融合决策层输出缓冲区。若碰撞确认标志=1，则通过回调接口立即通知APA应用层执行紧急停车（不等待下一周期）。

### 7.2 三层 AI 任务调度管理

#### 7.2.1 调度器设计原则

SDK 内部调度管理模块采用**静态优先级抢占式调度**模型，将所有AI相关任务按优先级分为四个调度级别，高优先级任务可抢占低优先级任务：

| 调度级别 | 任务类型 | 优先级 | 时间确定性要求 |
|---------|---------|--------|-------------|
| **级别1（最高）** | 碰撞中断响应、ESI DMA完成ISR | 硬实时 | 必须在<0.1ms内完成（ISR本身，不含AI计算） |
| **级别2** | 第1层MLP快速融合、碰撞初步确认后的CNN确认任务启动 | 软实时 | 每周期必须在T=45ms前完成 |
| **级别3** | 第2层-A 1D-CNN分类推理、OGM贝叶斯更新 | 尽力服务 | 在当前控制周期内完成，允许延迟至下一周期但须标注"延迟更新" |
| **级别4（最低）** | 第3层车位检测、卡尔曼滤波更新、诊断数据采集 | 后台任务 | 每2~5个控制周期完成一次，不得影响级别1~3任务 |

调度器在每个控制周期开始时（T=0ms）重置任务完成状态，并根据上一周期的执行情况动态调整本周期的任务列表（例如：若上周期某路1D-CNN因超时未完成，本周期将该路推理任务升至级别2优先执行）。

在裸机（Bare Metal）运行环境下，调度器通过循环主函数+中断抢占方式实现；在 AUTOSAR OS 环境下，调度器直接映射到 AUTOSAR 任务/事件机制（OsTask和OsAlarm），级别1映射为ISR Category 2，级别2~4映射为不同周期的OsTask，由AUTOSAR OS的Rate Monotonic调度算法保证执行顺序。

#### 7.2.2 第1层调度（每周期必执行）

第1层MLP快速融合是所有AI任务中优先级最高、实时性要求最严格的任务。调度规则如下：

**触发条件**：所有前6路（或后6路，取决于当前半周期）传感器的EDI响应帧均已接收完毕（通过DMA完成中断计数确认），或等待超时（超时时限15ms，超时时将已接收到的EDI帧用于推理，超时未到的路标记为本周期无效）。

**执行约束**：第1层任务从触发到推理完成的时间上限为2ms（含特征向量组装+MLP推理+规则引擎判断+输出写入），若超过2ms则触发超时告警并使用上一周期的输出结果（标记为"陈旧输出"），保证APA控制环不被阻塞等待。

**输出发布**：推理完成后，第1层通过共享内存缓冲区（无锁双缓冲，写入备缓冲后原子切换指针）将输出数据发布给APA应用层，保证APA读取时不会读到正在写入的不完整数据。

#### 7.2.3 第2层调度（按需触发）

第2层任务（1D-CNN）按需触发，不在每个控制周期强制执行，以避免在无深度分类需求时浪费算力。

**1D-CNN触发决策时机**：在第1层MLP完成（T≈40ms）后，由传输策略管理模块立即执行触发决策逻辑（见第4.4.3节），决策结果为：本控制周期对哪些路触发包络传输。决策在<1ms内完成，随即向ESI驱动发送包络上传命令。

**最大并发路数控制**：调度器限制同一控制周期内最多同时运行2路1D-CNN推理（防止工作内存区竞争）。若触发路数超过2路，第3路及以后排入下一控制周期队列，确保推理内核的工作内存不发生覆盖冲突。

**碰撞确认任务的优先权**：当碰撞初步标志=1且第二阶段CNN确认被调度时，具有抗占正在进行的普通1D-CNN推理的权利（因为碰撞确认的时间窗口更紧迫），当前被中止的推理排入下一周期重新执行。

#### 7.2.4 第3层调度（低频周期）

第3层场景理解任务（OGM更新、车位检测、卡尔曼滤波）以2~5个控制周期为触发间隔，由调度器根据当前算力余量自动计算最优触发间隔。

**算力余量评估方式**：调度器在每个控制周期结束时，将本周期实际消耗的CPU时间（通过高分辨率计时器测量）与100ms总预算相比较，计算算力占用率。若过去5个周期的平均占用率<60%，下次第3层触发间隔设为2个周期；若平均占用率>80%，触发间隔延长至5个周期；80%~60%之间取3个周期。

**第3层任务内部执行优先级**：第3层任务内部按以下顺序执行，前项完成后再执行后项，允许在任何项之前被第1层任务抢占（重新调度时从被抢占项继续）：
1. OGM坐标变换（依赖里程计，必须每次执行）
2. 包络投影+贝叶斯更新（依赖新到包络数据）
3. OGM老化衰减（每5次执行触发一次，不每次都执行）
4. 卡尔曼滤波状态更新（依赖第1层距离输出）
5. 车位角点检测（依赖历史距离序列，计算量最大，放最后）

### 7.3 超时处理与降级策略

#### 7.3.1 各任务超时定义与响应

| 任务 | 正常完成时限 | 超时阈值 | 超时响应 |
|------|-----------|---------|---------|
| 第1层MLP推理 | <2ms | 4ms | 使用上一周期输出，输出"陈旧"标志；连续3次超时触发级别2告警 |
| 包络DMA接收（单路） | <12ms | 20ms | 取消本次包络传输，本路本周期无包络数据；记录超时事件 |
| 1D-CNN推理（单路） | <5ms | 8ms | 中止推理，保留缓存表中上次有效结果；该路标记"推理超时" |
| 第3层OGM更新 | <30ms | 50ms | 跳过本次OGM更新，保留上次OGM地图；触发级别3告警 |
| ESI节点EDI响应 | <15ms/路 | 40ms | 该路本周期标记无效（见第3.6.2节超时处理规范） |

超时事件统一写入 SDK 内部的诊断事件日志（环形缓冲区，最多记录256条事件），每条事件包含：事件类型、触发时间戳、超时任务标识、累计触发次数。诊断工具可通过 UDS 0x22 服务读取事件日志，辅助性能调优和故障排查。

#### 7.3.2 系统级降级模式

SDK 定义以下三个系统级降级模式，当硬件故障或算力不足时自动激活，以尽量维持核心感知能力的可用性：

**降级模式1（轻度降级）**：1~2路传感器故障，其他路正常。
- 第1层：故障路特征向量置零，MLP推理继续运行，故障路有效性输出强制为0
- 第2层：故障路不触发包络传输，不进行1D-CNN分类
- 第3层：OGM在故障路覆盖区域的概率更新暂停（保留历史），老化衰减继续运行以防止虚假障碍物积累
- 输出置信度：降低10%（在综合置信度计算中乘以0.9的故障折扣系数）
- APA响应建议：继续运行，在故障路覆盖的方向增加安全裕度（默认+5cm）

**降级模式2（中度降级）**：同侧3~4路传感器故障，或ECU算力严重不足（占用率持续>90%）。
- 第1层：继续运行（EDI数据几乎不占算力），输出受限区域的有效性声明为低可信度
- 第2层：暂停1D-CNN包络传输触发（节省带宽和算力）
- 第3层：停止车位角点检测和卡尔曼轨迹预测；OGM每5个控制周期更新一次（最低频率）
- 输出置信度：降低30%
- APA响应建议：限制在简单场景（宽敞停车位）下运行；向驾驶员发出"传感器部分故障"提示

**降级模式3（严重降级）**：同侧加5路及以上传感器故障，或ECU算力持续严重不足。
- SDK上报严重故障告警（告警级别：CRITICAL）
- 所有AI模块降级为仅基于EDI标量数据的规则阈值模式（无神经网络推理）
- 感知能力退回至类AK1水平（仅距离阈值判断），APA系统不应在此模式下执行自动泊车
- 建议APA应用层退出自动泊车模式并强制要求驾驶员介入

降级模式的切换由调度管理模块根据节点状态机的实时故障统计自动触发，无需外部命令。降级模式恢复（从高降级级别恢复到低降级级别）须满足：故障传感器连续5次成功响应后才可降级解除，防止频繁的故障-恢复抖动。

#### 7.3.3 看门狗（Watchdog）集成接口

SDK 提供看门狗喂狗接口（`SDK_WatchdogFeed()`），调度器在每个控制周期完成第1层AI输出发布后调用一次该接口，向外部看门狗计时器发送喂狗脉冲，证明SDK的实时调度路径运行正常。若SDK因死锁、内存越界或异常卡死导致第1层超时未完成，则该喂狗接口不会被调用，看门狗定时器超时后触发ECU复位，实现系统级的软件故障自恢复。

喂狗接口的实现由 BSP 层提供（调用具体 ECU 平台的硬件看门狗驱动），SDK 仅定义接口签名和调用时序，不依赖特定的看门狗硬件实现。看门狗超时时间建议设置为200ms（两个控制周期），确保单次第1层偶发超时不触发复位，但连续两次超时（可能指示严重故障）必然触发复位。

### 8.2 算力需求规格（各模块 MFLOPS）

本节以每个控制周期（100ms）为单位，统计 SDK 全链路的 CPU 算力消耗，分析峰值算力需求和平均算力需求，为目标 ECU 芯片的算力选型提供依据。

#### 8.2.1 各模块算力估算

以下为各模块在一个控制周期内的 CPU 计算量（以 MFLOPS/周期 表示，假设目标 ECU 算力 400MFLOPS 即 Cortex-R52@400MHz 的近似有效算力）：

| 模块 | 计算量（MFLOPS/次） | 每周期执行频率 | 每周期消耗（MFLOPS） | 占400MFLOPS的比例 | 备注 |
|------|-----------------|------------|-----------------|---------------|------|
| ESI DMA接收+EDI解析 | 约0.01 | 每周期12次 | 约0.12 | <0.1% | 纯内存操作，几乎不占算力 |
| 包络预处理（截取+归一化，单路） | 约0.05 | 每周期0~4次 | 约0~0.2 | <0.1% | 仅触发传输时计算 |
| 第1层 MLP 推理（216→128→64） | 约0.4 | 每周期1次 | 约0.4 | 约0.1% | 最轻量，最高实时性 |
| 第1层 规则引擎判断 | 约0.01 | 每周期1次 | 约0.01 | <0.1% | 条件判断，几乎无计算量 |
| 第2层-A 1D-CNN推理（单路） | 约1 | 每周期0~4次 | 约0~4 | 约0~1% | 按触发路数线性累加 |
| 整体调度+时间戳测量 | 约0.02 | 每周期多次 | 约0.02 | <0.1% | 系统开销 |

#### 8.2.2 典型场景算力分析

以下三种典型场景涵盖泊车过程中的主要工况，对应不同的算力消耗水平：

**场景A：停车场巡航寻位（常态无碰撞，低频包络触发）**

此场景下车辆以低速（5~15km/h）在停车场行驶，第2层包络触发频率约为每周期1~2路，OGM每2~3周期更新一次：

| 汇总 | 算力消耗 |
|------|------|
| 第1层（必执行） | 约0.4 MFLOPS |
| 第2层-A（1~2路/周期） | 约1~2 MFLOPS |
| OGM+车位检测（每2~3周期） | 约1~1.5 MFLOPS（折算） |
| 卡尔曼+融合+系统开销 | 约0.3 MFLOPS |
| **场景A合计** | **约 2.7~4.2 MFLOPS**，占400MFLOPS约0.7~1.1% |

**场景B：泊车入位（密集障碍，高频包络触发）**

泊车执行阶段，前后方向均检测到目标，每周期触发3~4路包络传输，第3层每2周期执行一次：

| 汇总 | 算力消耗 |
|------|------|
| 第1层 | 约0.5 MFLOPS |
| 第2层-A（3~4路/周期） | 约3~4 MFLOPS |
| OGM+车位检测（高频） | 约1.5~2 MFLOPS（折算） |
| 卡尔曼+融合 | 约0.5 MFLOPS |
| **场景B合计** | **约 5~7 MFLOPS**，占400MFLOPS约1.3~1.8% |

**场景C：碰撞事件（峰值算力时刻）**

在碰撞初步标志置1的環境下，第二阶段CNN确认与正常第1层推理形成峰値算力需求：

| 汇总 | 算力消耗 |
|------|------|
| 第1层 | 约0.4 MFLOPS |
| 第二阶段 1D-CNN确认 | 约1 MFLOPS |
| 融合决策+系统开销 | 约0.1 MFLOPS |
| **场景C瞬时峰値** | **约 1.5 MFLOPS**，持续约15ms |

**算力安全余量评估**：最不利场景下（场景B叠加碰撞确认），瞬时峰値约8.5 MFLOPS，占Cortex-R52@400MHz有效算力（约400 MFLOPS）的2.1%。即使考虑内存访问等待引入的实际效率折损（约50%，即等效200 MFLOPS），峰値占用也仅约4.25%，算力储备充裕。

### 8.3 INT8量化后资源精简规格

本节对比 float32 未量化部署与 INT8 量化部署在内存、算力、精度三个维度的差异，作为量化必要性的依据，并给出在不同硬件条件下选择量化策略的建议。

#### 8.3.1 量化前后对比总表

| 指标 | float32 部署 | INT8 量化部署 | 提升倍数 | 备注 |
|------|------------|------------|--------|------|
| 模型权重 Flash 占用（必选三模型合计） | 约2,400KB | 约600KB | **4×** | — |
| 推理工作 SRAM（动态复用区峰值） | 约120KB | 约30KB | **4×** | 激活值从float32→INT8 |
| 第1层 MLP 推理时间 | 约4ms | 约1ms | **4×** | Cortex-R52@400MHz实测估算 |
| 第2层-A 1D-CNN 推理时间（单路） | 约20ms | 约5ms | **4×** | — |
| 第1层分类精度（准确率） | 基准（100%） | 约98%~99% | 损失约1~2% | PTQ方案，实测数据 |
| 第2层-A 分类精度 | 基准（100%） | 约97%~99% | 损失约1~3% | PTQ方案 |

> **说明**：以上推理时间为在 Cortex-R52 @ 400MHz 单核上的理论估算值，实际值受编译器优化、内存访问延迟（cache miss率）、中断响应占用等因素影响，通常在估算值的0.8~1.5倍范围内波动。实际项目应在目标 ECU 上通过 SWD 调试工具实测各模块的实际推理时间。

#### 8.3.2 量化策略选择建议

**目标ECU RAM ≥ 256KB、Flash ≥ 2MB（中高端平台，如NXP S32K3系列）**：
- 推荐采用完整 INT8 量化方案，部署必选三模型及第3层2D-CNN
- Flash 消耗约600KB（含双分区），SRAM 约70KB，均在硬件限制内
- 可选择将模型权重保存在外部SPI Flash以节省片内Flash

**目标ECU RAM 128~256KB、Flash 1~2MB（主流APA ECU）**：
- 推荐部署必选两模型（第1层MLP+第2层-A 1D-CNN）
- 第3层2D-CNN视Flash余量决定是否部署（Flash余量≥500KB时可单分区部署）
- OGM精度可降至10cm以减小SRAM占用（OGM双缓冲从38KB降至约10KB）
- SRAM 总占用约40~50KB，Flash 约500KB

**目标ECU RAM <128KB、Flash <1MB（低成本平台）**：
- 仅部署第1层MLP和第2层-A 1D-CNN，碰撞确认降级为纯规则阈值判断（超近距离<0.10m触发）
- 不部署OGM和第3层，感知能力回退至类AK1增强版（有障碍物类型分类但无场景理解）
- SRAM 约25KB，Flash 约200KB

#### 8.3.3 性能验收测试规范

SDK 在目标 ECU 部署后，须对以下关键性能指标进行实车或台架测试验收，验收结果须记录在项目测试报告中：

| 验收项目 | 测试方法 | 通过标准 |
|---------|---------|---------|
| 第1层推理实际延迟 | SWD硬件断点计时，连续记录100个控制周期的推理耗时 | 平均值<1.5ms，P99（99th percentile）<2ms |

## 附录B：关键工程注意事项

### B.1 传感器安装与标定注意事项

**安装位置精度要求**：传感器安装坐标须通过整车测量基准点（如前轴中心）精确测量，位置误差应小于5mm，方向角误差小于1°。

**ESI 总线指定后需进行全路通信自棃：**在 SDK 初始化完成后，建议对所有传感器节点进行端到端通信延迟测量，确认各路 EDI 响应延迟均在预期范围内。

**轮胎气压影响**：传感器安装高度随悬架状态变化，轮胎气压偏低时安装高度降低约5~15mm，会影响地面杂波的回波时延特性。建议在标定时将轮胎气压调整到标准值±5%范围内。

### B.2 ESI 总线电气完整性注意事项

**总线电阻匹配**：ESI 总线（基于DSI3）对终端电阻匹配有严格要求。若总线末端未正确终端匹配，高速数据帧（包络传输阶段的1kbps帧）可能因反射导致误码率上升，表现为包络数据中出现随机的比特翻转。SDK 的 EDI 帧 CRC 校验会检测到帧级别的错误，但包络序列传输通常不带帧级 CRC，位翻转可能被直接当作有效数据传入推理引擎。建议在 PCB 设计阶段对 ESI 总线进行眼图测试，验证信号完整性。

**多路传感器同步测量串扰**：DJ628.27 的超声波测量在同一时刻只允许相邻传感器之间互不干扰。若 ESI 总线主机同时向距离过近的传感器发送 Chirp 触发，相邻传感器的发射声波可能直接被旁边传感器接收（串扰），导致虚假近距离回波（距离约等于相邻传感器间距/2）。SDK 通过交替触发（前6路/后6路50ms交错）来规避大部分串扰，但同排相邻传感器之间（如前保险杠5个传感器）的串扰须通过合理的触发时序配置（让距离最近的相邻传感器错开100ms触发）进一步抑制。

### B.3 AI 模型版本管理注意事项

**训练数据与部署环境的一致性（Domain Shift）**：最常见的部署性能下降原因是训练数据与部署环境之间的分布差异。当 SDK 从一种车型移植到另一种车型时，即使传感器型号相同，传感器安装角度差异导致的包络形态差异，都可能使原有模型的精度明显下降。建议在每次车型迁移时，至少采集300~500条新车型的数据用于微调（Fine-tuning），而非直接没用原车型的模型。

**量化参数与训练数据的绑定关系**：INT8 量化参数（缩放因子和零点）是在特定的训练/校准数据集上计算得出的，与模型权重紧密绑定。若更换了训练数据集（即使网络结构不变），必须重新进行量化校准，不得沿用旧的量化参数。SDK 中每个量化模型文件的文件头包含校准数据集的摘要哈希值，工具链在量化转换时自动检验参数匹配性；若哈希不匹配，工具链拒绝生成量化代码。

**模型版本号与 SDK 固件版本的兼容性**：SDK 在初始化时读取 Flash 中各模型的元数据，检查模型主版本号是否与 SDK 当前接口版本兼容（见附录A接口格式）。主版本号不匹配时 SDK 拒绝加载该模型，并将健康字中的"模型不兼容"位置1，APA 控制模块须据此禁止自动泊车功能。

### B.4 软件安全与 AUTOSAR 合规注意事项

**MISRA-C 合规检查覆盖范围**：SDK 核心代码（推理引擎、融合决策、调度管理）须通过 MISRA-C 2012 全量检查，Required 规则不得有违规（Advisory 规则可豁免并附说明文档）。AI 模型推理工具链生成的 C 代码须至少通过 Required 规则。HAL 层代码（BSP 相关）可豁免部分 MISRA 规则，须在豁免文档中记录原因。

**堆栈溢出防护**：SDK 使用的中断服务例程（ISR）须通过静态分析工具（如 StackAnalyzer）验证最坏情况栈深度，确保不超过为中断栈分配的空间（建议为最坏情况分析值的1.5倍）。AI 推理工具链生成的推理 C 代码须验证无递归调用（标准 1D-CNN 推理无递归，但须确认工具链实现）。

**NVM 写入磨损保护**：SDK 在每次 OTA 更新或模型 CRC 校验失败后执行 NVM 写入，须使用磨损均衡算法，避免对同一 Flash 扇区反复擦写。建议为 SDK 配置参数区划分至少4个镜像扇区进行滚动写入（每个扇区最少10,000次擦写寿命下，4扇区滚动可提供至少40,000次逻辑写入次数，满足整车生命周期内的升级需求）。
